Overview

purpose

The event resource provides information about road events (accidents, constructions, special events). The events resource is a list of event elements matching the filtering parameters if any are provided.
methodGET

Parameters

In addition to the filters described below, the events resource also supports format selection and language negotiation.

By default, all fields of the event structure can be used as filtering criteria. So for example, a query to retrieve events on road HWY-73 will just contain ...?road_name=HWY-73. Also, the following parameters are supported

ParameterRequired/optionalDescription
startDateoptional

The start date parameter allows for searching events that will be or have been active anytime after the provided value. The start date parameter can be past or future. The parameter should be complete DateTime. Issue 7
endDateoptional

The end date parameter allows for searching events that will be or have been active anytime before the provided parameter. The end date parameter can be past or future. The parameter should include complete DateTime including the timezone.
statusoptional

By default the API should only send active events. If the client wants to retrieve archived events, it should provide the status with the value archived or a star * to retrieve both active and inative events. Issue 7
severityoptional

Although it is possible to query events for a given severity, it is also possible to query all events below or equal level 2 with severity=2- or above or equal to level 2 severity=2+ issue 2
jurisdictionoptional

Clients should be able to request events for a given jurisdiction on multi-jurisdiction endpoints or on agregators. The method still has to be defined : issue 3
geographyoptional

Filtering near a point or linestring or within a polygon. The method still has to be defined : issue 4

Data structure

The following table provide the general message structure for the events resource. Events is a collection of the event structure.

FieldTypeDescription
selfLink

Mandatory

Self link to the current resource. In XML, the rel attribute needs the value self.
upLink

Mandatory

Link to the parent resource. In XML, the rel attribute needs the value up
eventsCollection of Event

Mandatory. Can contain multiple event elements, at least one occurrence is mandatory

Main content of the events resource, a collection of event elements.
paginationPagination structure

Conditional (mandatory if more than one page)

The pagination structure provides information about the current position in the pagination and links to other pages

— offset

integer

Optional

Offset used to access the current page

— limit

integer

Optional

Number of events listed in the current page

— next

Link

Conditional (mandatory if a next page exists)

Atom link pointing to the next page of the event. Relation attribute value is next

— previous

Link

Conditional (mandatory if a previous page exists)

Atom link pointing the previous page of event. Relation attribute value is previous

Event structure

The event structure is the main element of the events collection. Although an event should be considered as independent, it is possible that one major event (mainly construction) could be split across several events

issue 1

FieldTypeDescription
selfLink

Mandatory

Self link to the current resource. In XML, the rel attribute needs the value self.
jurisdictionLink

Mandatory

Link to the jurisdiction which published the event. In XML, the rel attribute needs the value jurisdiction.
statusEnum

Mandatory

Status of the event. The status allows a client to determine if the current event should be considered as currently effective. Value list:

- active : The event should be considered as effective
- archived : The event should not be considered as effective
headlineFree text

Mandatory

Title of the event. Should be shorter than 500 characters.
eventTypeEnum

Mandatory

Basic type of event. Value list

- construction :  planned road work
- event : special events (fair, sport event, etc.)
- incident : accidents and unplanned road work.
eventSubTypeEnum

Optional

List of values to provide more detail about the type of event.
severityEnum

Mandatory Issue 16

Value list to be defined, either a scale (1 to N) or value list with defined semantics (Extreme/Severe/Moderate/Minor/Unknown) Issue 2
certaintyEnum

Optional

Degree of certainty of the event. Should only be used for unplanned events (eventType = incident). Value list to be confirmed. Could be observed/likely/possible/unknown

creationDateDateTime

Mandatory

Date of creation of the event element
lastUpdateDateTime

Mandatory

This field provides the date and time of the last update. The value is the same as the creation date if no change occurred.
relatedEventLink

Optional

This field is used to group events together. In specific situations (for example complex construction projects), several events might be related together. This field should be used to point a related event. In XML, the rel attribute needs the value related
detourFree text

Optional

If some detour information is available, it could be provided in this field as free text Issue 9
travelerMessageEnum

Optional

In order to simplify analysis by automated tools, this field allows for providing predefined messages with a known semantic. For example resident-only means that the circulation is limited to people living on this street. The exact enum still has to be determined.Issue 17
geometrygeography field

Mandatory

Geographical information about the event using the GML notation. Valid type of geometry can be points, multipoints, linestring, multilinestrings and polygon.

roadsCollection of road

Mandatory, support multiple occurrences of road element.

List of roads affected by the current event. One event can impact several roads.
scheduleschedule structure

Mandatory

Complex item to provide time related information
attachmentsCollection of attachment

Optional, support multiple occurrences of attachment element.

Collection of attachments provided additional information about the event (PDFs, images, etc.)

Road structure

The road data format is used to provide information about the road(s) affected by an event.

FieldTypeDescription
roadNameFree text

Mandatory

Name of the road affected by the event. When possible, this field should contain a value that fits the national symbology for roads and highways. Issue 11
fromfree text

Optional

Approximate start point of the event on the road. It can be an intersection with another street or it can be a mileage indication.

This value should not be used to determine the exact start point since it can be an approximation. The geometry field should be considered as the reference for exact location.

If no to field is provided, the from can be considered as a "near".
tofree text

Optional

Approximate end point of the event on the road. It can be an intersection with another street or it can be mileage indication
directionEnum

Conditional

Direction of the road that is affected by the event.

If a lane_status is provided, the direction becomes mandatory. In the same situation, if both directions are affected, two occurrences of the road element are needed, one for each direction and with a dedicated lane_status for each direction.
laneStatusbitmap

Optional

Bitmap representing the lanes that are closed because of the event. For example 0110 means that the 2 central lanes of a 4 lanes roads are closed.

The lane status is applied to direction provided
cityfree text

Optional

City in which the event is taking place (or near)
impactedSystemsCollection of impactedSystem

Optional

Allows provision of information about other systems that can be affected

— impactedSystem

Enum

Optional

Value list of systems:

ROAD
SIDEWALK
BIKELANE
ONSTREET-PARKING
PARKING
restrictionsCollection of restriction

Optional

Some events may come with some restrictions in terms of vehicles using the road (speed, weigth). The restriction structure provides a way to provide the restrictions

— restriction

Enum

Optional

Each restriction element provides a limitation on the current road element. The node element can be speed, weight, length, width. A unit can provide the unit used to measure the restriction (t for tons, lb for pounds, etc.).

Detail for the restrictions supported and formatting to be determined Issue 18

Schedule structure

FieldTypeDescription
timezonetimezone

Optional

Timezone to be used for this event. Timezone abbreviation (e.g EST) or the continent/city timezone code America/Montreal
startDateDate

Mandatory

Start date of the event.
endDateDate

Optional

End date of the event. If no end date is provided, the event should be considered continuing forever.
dayOfWeekEnum

Optional

Comma separated list of days of the week where the event should be considered as active. For example wednesday, thursday, friday would be 3,4,5. The days are Monday to Sunday from 1 to 7.
timestructure

Optional

Provide detail about the specific times of the event. Multiple occurrences of the time structure are allowed so specify multiple timeframes in a day.

— startTime

Time

Optional

Start time of the event. If the event is spread across several days, the start time is to be applied for each day

— endTime

Time

Optional

Start time of the event. If the event is spread across several days, the start time is to be applied for each day. end_time is mandatory if a start_time is provided.
specificDatesCollection of specificDate

Optional

Since some days might not follow the regular schedule, it is possible to provide a schedule for individual days (for example vacations). Issue 19
